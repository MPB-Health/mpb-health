## Hotfix — 2026-05-18 — Leads visibility recovery + Leads list default sort

User-visible symptom: the Leads list was missing brand-new submissions and "the most recent leads" only showed up in the dashboard Recent Leads widget. Root cause was two compounding bugs — one in the public intake RPC, one in the list default sort.

- Added migration [`20260620590000_crm_lead_last_activity_default_sort.sql`](../../../supabase/migrations/20260620590000_crm_lead_last_activity_default_sort.sql) — adds `lead_submissions.last_activity_at` generated column = `COALESCE(last_touched_at, created_at)` plus an org-scoped DESC NULLS LAST index. Existing strict "Last Touched" semantics (rep-initiated only, per Phase 1 trigger) remain intact on `last_touched_at`. The list now defaults to `last_activity_at` so freshly-created leads with no rep touch yet surface at the top by `created_at`; worked leads keep sorting by their most recent rep activity. Click the "Last touched" column header to fall back to the strict rep-initiated sort.
- [`packages/crm-core/src/leads/leadService.ts`](../../../packages/crm-core/src/leads/leadService.ts) `LEAD_ROW_SELECT` now includes `last_activity_at`; `getLeads` defaults `.order()` to `last_activity_at` desc nullslast.
- [`packages/crm-core/src/leads/leadTypes.ts`](../../../packages/crm-core/src/leads/leadTypes.ts) `Lead` gained `last_activity_at`; `LeadFilters['sortBy']` accepts `'last_activity_at'`.
- [`apps/crm/src/pages/LeadsList.tsx`](../../../apps/crm/src/pages/LeadsList.tsx) initial `listSort` is `{ by: 'last_activity_at', dir: 'desc' }`. The "Last touched" header tooltip explains the default vs strict-touch behavior.
- Added migration [`20260620600000_crm_lead_intake_default_org.sql`](../../../supabase/migrations/20260620600000_crm_lead_intake_default_org.sql) — fixes the bug that made every public website submission since `2026-04-29` invisible to non-admin reps:
  - The `submit_public_lead(payload jsonb)` RPC (used by every form on apps/website) deliberately blocks anonymous callers from setting CRM-internal attribution columns including `org_id`. It also never supplied a default, so every public submission landed with `org_id = NULL` and was filtered out by RLS policy `org_leads_select` (`org_id IS NOT NULL AND is_org_member(org_id)`).
  - Seeds a non-sensitive `system_settings` row `crm.intake_default_org_id` (= `00000000-0000-4000-a000-000000000001`). Operators can repoint this without a code deploy if a multi-tenant cutover ever happens.
  - Recreates `submit_public_lead` to read that setting and stamp `org_id` on every insert. Anonymous callers still cannot override the value.
  - Backfills `org_id = MPB Health` on the **230** existing NULL-org leads so reps regain visibility immediately. Per the design decision (Option `none` for retro emails), the backfill deliberately does NOT enroll those leads in the Quote Response cadence or send Email #1 — months-late auto-responses would do more harm than silence.
